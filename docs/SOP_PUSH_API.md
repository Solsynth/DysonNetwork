# SOP Push API (Solar Network Push)

This document describes the SOP push provider in Ring and how clients should integrate with it.

## Overview

SOP is a Ring-native notification delivery path.

It uses:

- a server-generated SOP token for device registration
- a notification list API for initial sync / missed notifications
- an SSE stream API for real-time delivery

Unlike Apple Push Notification Service, Firebase Cloud Messaging, or UnifiedPush, SOP does not require the client to provide a third-party push token.

## What SOP Actually Does

SOP is not a provider push gateway.

It is a Ring-side delivery mechanism built from:

- subscription registration
- account-scoped notification listing
- account-scoped server-sent event streaming

In practice, a healthy SOP client should:

1. authenticate with normal Dyson auth
2. register SOP for the current device/session
3. fetch notification history / current unread state from the SOP list API
4. keep an SSE connection open to receive new notifications in real time
5. reconnect and re-sync on disconnect

## Provider Name

`PushProvider.Sop`

## Important Delivery Behavior

SOP has device-aware deduplication behavior.

When Ring sees an active SOP stream for a device, that device may be treated as already covered by SOP delivery for real-time notification fanout. Clients should therefore not assume that another real-time path on the same device will always duplicate the same event.

In short:

- if SOP is connected and healthy, the client should receive real-time notifications from SOP
- if SOP is broken on the client side, the client may appear to "miss" notifications until it re-syncs from the list API
- because of that, clients must treat the list API as the source of truth and SSE as the live-update channel

## Authentication Models

SOP uses two auth modes depending on the endpoint.

### Bearer-auth endpoints

These require normal Dyson user auth:

- `POST /api/notifications/sop/subscription`
- `GET /api/notifications/subscription`
- `GET /api/notifications/subscription/current`
- `PUT /api/notifications/subscription`
- `DELETE /api/notifications/subscription/{subscription_id}`

### SOP-token endpoints

These use the SOP token returned by registration:

- `GET /api/notifications/sop`
- `GET /api/notifications/sop/stream`

## SOP Token Transport

For SOP-token endpoints, the token can be provided in any one of these forms:

- query string: `?token=<sop_token>`
- header: `X-SOP-Token: <sop_token>`
- header: `Authorization: SOP <sop_token>`

## Endpoint Reference

## 1. Register SOP Token

Registers or updates the SOP subscription for the current authenticated session and returns a server-generated SOP token.

- **Method:** `POST`
- **Path:** `/api/notifications/sop/subscription`
- **Auth:** Bearer auth
- **Content-Type:** `application/json`

### Request Body

```json
{
  "device_name": "My iPhone",
  "app_id": "com.example.app"
}
```

### Request Fields

- `device_name` - optional human-readable device name
- `app_id` - optional app identifier used by Ring notification app scoping

### Response Example

```json
{
  "token": "2ff9...",
  "subscription": {
    "id": "5c6ec8f6-3bc1-4de7-8ed6-9f9268ef642f",
    "account_id": "f8f6f6d3-2f64-44e8-b377-66f8935a54a7",
    "device_id": "<session-client-id>:sop",
    "device_token": "2ff9...",
    "provider": 2,
    "created_at": "2026-03-09T05:40:00Z",
    "updated_at": "2026-03-09T05:40:00Z"
  }
}
```

### Notes

- the SOP token is generated by the server
- the returned `subscription.device_id` is the current session client id with `:sop` suffix
- clients should persist the SOP token securely
- clients should re-register when the authenticated session changes or when they need a fresh SOP token

### Example

```bash
curl -X POST "https://ring.example.com/api/notifications/sop/subscription" \
  -H "Authorization: Bearer <access_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "device_name": "My iPhone",
    "app_id": "com.example.app"
  }'
```

## 2. List Notifications via SOP Token

Lists notifications for the account bound to the SOP token.

This endpoint is intended for:

- initial notification sync after login
- backfilling notifications missed while offline
- recovering from SSE disconnects
- confirming current unread / recent state

- **Method:** `GET`
- **Path:** `/api/notifications/sop`
- **Auth:** SOP token
- **Query Parameters:**
  - `offset` - default `0`
  - `take` - default `8`
  - `app` - optional app scope

### Important Behavior

This endpoint does **not** mark notifications as viewed.

That is different from the normal authenticated notification list endpoint:

- `GET /api/notifications` marks returned notifications viewed unless `unmark=true`
- `GET /api/notifications/sop` never auto-marks viewed

### Response Headers

- `X-Total` - total notification count visible to this SOP token/account/app scope

### Example

```bash
curl "https://ring.example.com/api/notifications/sop?offset=0&take=20&token=<sop_token>"
```

### Header Example

```bash
curl "https://ring.example.com/api/notifications/sop?offset=0&take=20" \
  -H "X-SOP-Token: <sop_token>"
```

### Response Shape

Returns a JSON array of `SnNotification` objects.

Typical fields include:

- `id`
- `account_id`
- `topic`
- `title`
- `subtitle`
- `content`
- `meta`
- `priority`
- `app_id`
- `push_type`
- `created_at`
- `updated_at`
- `viewed_at`

### Client Expectations

Clients should:

- use this endpoint on startup
- use this endpoint after SSE reconnect
- de-duplicate by notification id
- not assume the SSE stream alone is sufficient as the sole source of truth

## 3. Stream Notifications via SOP Token (SSE)

Opens a server-sent events stream for real-time notifications.

- **Method:** `GET`
- **Path:** `/api/notifications/sop/stream`
- **Auth:** SOP token
- **Content-Type:** `text/event-stream`

## SSE Event Types

### `ready`

Sent once after the stream is established.

```text
event: ready
data: {"status":"connected"}
```

A client should treat this as confirmation that the server accepted the SOP token and the SSE stream is alive.

### `notification`

Sent for each new notification delivered to the account.

```text
event: notification
data: { ...sn_notification_json... }
```

The `data` payload is a JSON-serialized `SnNotification`.

## SSE Client Requirements

A production client should:

- open the stream only after SOP registration succeeds
- wait for the `ready` event before treating the stream as active
- reconnect automatically on disconnect
- re-fetch from `GET /api/notifications/sop` after reconnect
- de-duplicate notifications by `id`
- handle duplicate delivery gracefully
- handle temporary disconnects, tab suspension, app backgrounding, and network transitions

## Browser Example

```javascript
const token = "<sop_token>";
const es = new EventSource(
  `/api/notifications/sop/stream?token=${encodeURIComponent(token)}`
);

es.addEventListener("ready", (evt) => {
  console.log("SOP stream ready", evt.data);
});

es.addEventListener("notification", (evt) => {
  const notification = JSON.parse(evt.data);
  console.log("new notification", notification);
});

es.onerror = (err) => {
  console.error("SOP stream error", err);
};
```

## Recommended Client Flow

### Startup / login flow

1. authenticate with normal Dyson auth
2. call `POST /api/notifications/sop/subscription`
3. persist the returned SOP token securely
4. call `GET /api/notifications/sop` for initial state
5. open `GET /api/notifications/sop/stream`
6. wait for `ready`
7. process incoming `notification` events

### Reconnect flow

1. detect SSE disconnect or failure
2. reconnect the SSE stream
3. after reconnect succeeds, call `GET /api/notifications/sop` again
4. merge results by notification id

### Background / resume flow

When the app resumes from background or the browser/tab becomes active again:

1. verify the SSE stream is still alive
2. if not, reconnect
3. re-sync from `GET /api/notifications/sop`

## Troubleshooting Checklist

If the client appears to miss real-time SOP notifications, check these first:

1. did SOP registration succeed?
2. is the client using the latest SOP token?
3. does the SSE request actually connect?
4. does the client receive the `ready` event?
5. does the SSE connection stay open over time?
6. does the client reconnect after disconnect?
7. after reconnect, does the client re-fetch from `GET /api/notifications/sop`?
8. is the client silently failing to parse `notification` event payloads?
9. is the client de-duplicating too aggressively and dropping valid events?
10. is the client scoping requests with the expected `app` value?

## Common Failure Modes

### Connected endpoint but no live updates

Possible causes:

- SSE connection closed and was not re-opened
- client never waited for or handled the `ready` event
- client parsing bug in `notification` event handler
- app lifecycle / browser lifecycle suspended the stream

### Notifications visible in list API but not shown live

This usually points to a client-side SOP stream problem rather than missing server persistence.

Check:

- SSE reconnect behavior
- event parsing
- local notification store merge logic
- app background / foreground transition logic

### SOP registered, but another push provider seems inactive

This can be expected depending on client/device behavior and server-side delivery selection. Clients integrating SOP should treat SOP as a primary real-time path and use the list API for recovery.

## Push Device Management

The regular notification controller exposes account-scoped push subscription management for all providers, including SOP.

## List Registered Push Devices

- **Method:** `GET`
- **Path:** `/api/notifications/subscription`
- **Auth:** Bearer auth
- **Query Parameters:**
  - `app` - optional app scope

Returns all registered push subscriptions for the current account, including SOP subscriptions.

## Get Current Device Active Subscription

- **Method:** `GET`
- **Path:** `/api/notifications/subscription/current`
- **Auth:** Bearer auth

Returns the effective active subscription for the current client device.

Selection rule:

- SOP on this device has priority over APNS/FCM or other providers
- if SOP is not registered on this device, the most recently updated subscription for the current device is returned

## Unregister by Subscription ID

- **Method:** `DELETE`
- **Path:** `/api/notifications/subscription/{subscription_id}`
- **Auth:** Bearer auth

Deletes the specified subscription if it belongs to the current account.

## Register Non-SOP Push Subscription

- **Method:** `PUT`
- **Path:** `/api/notifications/subscription`
- **Auth:** Bearer auth
- **Query Parameters:**
  - `force` - default `false`

### Request Body

```json
{
  "device_token": "<provider_token_or_endpoint>",
  "device_name": "My iPhone",
  "provider": 0,
  "app_id": "com.example.app"
}
```

Behavior when SOP is already registered on the current device:

- if `force=false`, the registration becomes a no-op and still returns `200 OK`
- the response body is the current active SOP subscription
- if `force=true`, the non-SOP registration proceeds normally

## Notes

- `PUT /api/notifications/subscription` does not accept `PushProvider.Sop`
- use `POST /api/notifications/sop/subscription` for SOP registration
- SOP tokens are currently persisted in push subscription storage and are treated as bearer credentials for SOP APIs
- clients should handle SOP tokens like secrets
- clients should prefer the list API as recovery truth and SSE as the live-update channel

# Multi Wallet

This document describes the wallet model update that adds:

- multiple personal wallets per account
- default wallet resolution for account-based flows
- realm-owned wallets for organization usage
- optional human-readable public wallet IDs
- wallet-targeted transfers

These changes are implemented in `DysonNetwork.Wallet` and the shared wallet proto/model.

## Overview

Wallets are no longer limited to one wallet per account.

The system now supports:

1. multiple wallets for one account
2. one default wallet for account-based compatibility
3. wallets owned by a realm for organization use
4. optional public wallet IDs for bank card-like user experience

Internal service calls still use the canonical wallet `Guid`.

The human-readable wallet ID is optional and nullable.

## Ownership Model

`SnWallet` now supports two owner shapes:

- personal wallet: `account_id` is set
- realm wallet: `realm_id` is set

Only one owner target should be used for a wallet.

### Wallet fields

Key wallet fields:

| Field | Type | Meaning |
| --- | --- | --- |
| `id` | `Guid` | Canonical wallet identifier |
| `account_id` | `Guid?` | Personal wallet owner |
| `realm_id` | `Guid?` | Realm wallet owner |
| `name` | `string` | User-visible wallet name |
| `is_primary` | `bool` | Owner default wallet marker |
| `public_id` | `string?` | Optional human-readable wallet ID |

## Default Wallet Behavior

Legacy account-based wallet flows still work.

Whenever the code resolves a wallet by `account_id`, the default wallet is selected using this order:

1. the wallet where `is_primary = true`
2. if none is marked primary, the first created wallet for that account

This preserves backward compatibility for older APIs and service flows that still think in terms of one wallet per account.

## Wallet Quota

Personal wallets currently use a service-side quota:

```text
3 wallets per account
```

Realm wallets do not currently use the same quota rule.

## Public Wallet ID

Wallets can expose an optional bank card-like ID for end-user transfers.

Example format:

```text
DNW-ABCD-EFGH-JKLM
```

### Properties

- generated by the server
- unique across wallets
- nullable by default
- enabled or disabled by explicit API call
- not used as the canonical internal identifier

## Personal Wallet APIs

### Get default wallet

**GET** `/api/wallets`

Returns the current user's default wallet.

This is the backward-compatible single-wallet endpoint.

### List all personal wallets

**GET** `/api/wallets/all`

Returns all personal wallets owned by the current user.

Current behavior only lists account-owned wallets for the current user.

### Get a specific wallet

**GET** `/api/wallets/{id}`

Returns a wallet when the current user:

- owns the personal wallet
- or has viewer access to the owning realm publisher for a realm wallet

### Create a personal wallet

**POST** `/api/wallets`

Request example:

```json
{
  "name": "Travel Wallet"
}
```

The first personal wallet for an account becomes the default automatically.

### Set default wallet

**POST** `/api/wallets/{id}/default`

Sets the specified personal wallet as the current user's default wallet.

### Enable public ID

**POST** `/api/wallets/{id}/public-id/enable`

Enables and generates a public wallet ID if one does not already exist.

### Disable public ID

**POST** `/api/wallets/{id}/public-id/disable`

Removes the public wallet ID from the wallet.

## Realm Wallet APIs

Realm wallets are organization wallets.

### Create a realm wallet

**POST** `/api/wallets`

Request example:

```json
{
  "name": "Treasury",
  "realm_id": "6f7413d0-44d8-4fd3-84b5-d2603fb0e5a2"
}
```

Authorization rule:

- caller must be an editor of the realm publisher

### Access rules

Realm wallet read access:

- viewer or above on the realm publisher

Realm wallet public ID management:

- editor or above on the realm publisher

Realm wallet spending:

- editor or above on the realm publisher

## Transfers

Transfers now support multiple target styles.

### HTTP endpoint

**POST** `/api/wallets/transfer`

Request supports:

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `currency` | `string` | Yes | Wallet currency |
| `amount` | `decimal` | Yes | Positive amount |
| `remark` | `string` | No | Custom remark on the main transfer transaction |
| `pin_code` | `string` | Yes | Present in request model |
| `payer_wallet_id` | `Guid?` | No | Explicit source wallet |
| `payee_account_id` | `Guid?` | No | Target account default wallet |
| `payee_wallet_id` | `Guid?` | No | Explicit target wallet |
| `payee_public_id` | `string?` | No | Target wallet by public ID |

At least one payee target must be provided.

### Transfer resolution rules

Payer resolution:

1. explicit `payer_wallet_id`
2. otherwise current user's default wallet

Payee resolution:

1. `payee_wallet_id`
2. `payee_public_id`
3. `payee_account_id` resolved to target account default wallet

### Transfer remark behavior

The custom transfer remark is stored on the main transfer transaction.

The separate fee transaction keeps its own generated fee remark.

## gRPC / Proto Changes

The wallet proto was expanded.

### `DyWallet`

Added:

- `realm_id`
- `name`
- `is_primary`
- `is_default`
- `public_id`

### `DyGetWalletRequest`

Added:

- `wallet_id`
- `public_id`

### `DyCreateWalletRequest`

Added:

- `realm_id`
- `name`

### `DyTransferRequest`

Added:

- `payer_wallet_id`
- `payee_wallet_id`
- `payee_public_id`
- `remarks`

## Compatibility Notes

### Preserved

- account-based callers still work through default wallet resolution
- `GET /api/wallets` still returns one wallet
- existing account-based transfer calls still work

### New behavior

- `GET /api/wallets/all` lists multiple personal wallets
- explicit wallet-to-wallet transfers are supported
- wallet public IDs can be used for transfers
- realm wallets can be created and spent by authorized organization members

## Migration Notes

Database migration added:

```text
DysonNetwork.Wallet/Migrations/20260504130000_AddMultiWalletSupport.cs
```

The migration:

- makes `account_id` nullable on `wallets`
- adds `realm_id`
- adds `name`
- adds `is_primary`
- adds `public_id`
- backfills existing account wallets to `is_primary = true`

## Current Limitations

1. personal wallet quota is hardcoded to `3`
2. realm wallet listing is not yet exposed as a separate list endpoint
3. `pin_code` is still present in transfer requests, but validation is not implemented here
4. app payout metrics are still separate from general wallet analytics

# Custom App Notifications

This document describes the custom app notification endpoint and the notification payload rules.

## Endpoint

Custom app notifications are sent through Padlock:

- `POST /api/private/apps/{appId}/notifications`

Authentication:

- `Authorization: Bearer {custom_app_api_key}`

The bearer token must be a valid custom app API key from Develop.

## Authorization rules

A notification is only delivered to accounts that:

- authorized the target custom app
- granted the `notifications.send` scope

This means a custom app can only send notifications to users that explicitly approved that capability.

## Targeting modes

The endpoint supports three targeting modes:

### Single account

```json
{
  "account_id": "00000000-0000-0000-0000-000000000001",
  "topic": "order.created",
  "title": "Order created",
  "body": "Your order was created."
}
```

### Multiple accounts

```json
{
  "account_ids": [
    "00000000-0000-0000-0000-000000000001",
    "00000000-0000-0000-0000-000000000002"
  ],
  "topic": "campaign.update",
  "title": "Campaign update",
  "body": "A new update is available."
}
```

### Broadcast to all authorized users

```json
{
  "broadcast_to_all": true,
  "topic": "maintenance",
  "title": "Maintenance notice",
  "body": "Scheduled maintenance starts soon."
}
```

## Request fields

Supported request body fields:

- `account_id`
- `account_ids`
- `broadcast_to_all`
- `topic`
- `title`
- `subtitle`
- `body`
- `action_uri`
- `push_type`
- `is_silent`
- `is_savable`
- `meta`

If `broadcast_to_all` is `false`, at least one of `account_id` or `account_ids` must be provided.

## Topic format

The client-provided topic is not used directly.

The system rewrites it to:

- `<publisher_name>.<app_slug>.<developer_topic>`

Example:

- developer topic: `order.created`
- app slug: `shop`
- publisher name: `Acme`
- final topic: `Acme.shop.order.created`

This prefix is generated by the system and cannot be overridden by the app.

## Meta behavior

The request may include a custom `meta` object.

Padlock preserves that object, but always injects a system-generated field:

- `sent_by_app`

Example final meta shape:

```json
{
  "foo": "bar",
  "sent_by_app": {
    "id": "00000000-0000-0000-0000-000000000123",
    "slug": "shop",
    "name": "Shop",
    "publisher": "Acme"
  }
}
```

Rules:

- `sent_by_app` is generated by the system
- clients should treat `sent_by_app` as trusted system metadata
- custom app code must not rely on overriding `sent_by_app`

## Delivery behavior

Padlock resolves the target account list from authorized app records.

Only accounts matching all of the following are included:

- `app_id == {appId}`
- authorization record is not deleted
- authorization type is OIDC
- saved scopes include `notifications.send`

If no eligible targets are found, the endpoint returns success with `sent = 0`.

## Response

Example response:

```json
{
  "sent": 12,
  "scope": "notifications.send",
  "broadcast_to_all": true
}
```

## Notes

- This endpoint is private infrastructure for first-party web/app integrations.
- Route ownership is under Padlock because delivery depends on user authorization state.
- The app resource path is aligned with the new private route layout: `/api/private/apps/{appId}/notifications`.
